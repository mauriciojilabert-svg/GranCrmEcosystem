**Goal:** migrate InciTrack from a server-rendered Django app (Duralux
templates, own navbar) into a **native React + Vite Module-Federation remote**,
mounted by the GranCRM shell exactly like **Call Reviews** — responsive,
portable, no iframe. The shell provides the chrome; InciTrack exposes `./App`
and renders inside it as a real SPA talking to a JSON API.

> **Explicit decision:** the iframe approach is **rejected** for InciTrack. It is
> the platform's *legacy* mode (`modo: iframe`), used today only by Panel SA and
> WSP. The target architecture for InciTrack is `modo: spa_remote`. Any earlier
> iframe-based plan in this folder has been deleted.

## The authoritative spec — read it first

`/home/admincrm/orquestador/docs/GUIA_INTEGRACION_APP_SATELITE.md` is the
platform's master integration guide. It defines the **mount contract**, the JWT
SSO, the multi-tenant rules, the `dios.json` shape, the nginx wiring, and a
full acceptance checklist (§2.7). **Every plan below references its sections
instead of duplicating them.** The executor must read the cited sections.

The reference implementation to copy from is **Call Reviews**
(`/home/admincrm/call_reviews/`), which the guide calls "el más correcto":
- Frontend: `call_reviews/frontend/` (React 18 + Vite, `@module-federation/vite`,
  `name: 'callreviews'`, `exposes: { './App': './src/App.tsx' }`, builds to
  `static/mf/callreviews`).
- API: **django-ninja** at `config/urls.py → path("api/v1/", api.urls)` →
  served under `/callreviews/api/v1/...`.
- Auth: `grancrm_auth/middleware.py` + `config/grancrm_session.py` +
  `utils/tenant_middleware.py` (the obligatory 3-middleware order, guide §1.6).
- Registration: `dios.json` with `modo: spa_remote` + MF fields.

## What is missing (the migration)

InciTrack has **no** `frontend/`, **no** JSON API (only a couple of ad-hoc
`JsonResponse` AJAX endpoints), and no django-ninja/whitenoise. It renders
everything via Django templates with its own Duralux navbar. The migration
builds the API + React remote and retires the templates.

## Known follow-ups (discovered during 001 execution)

- **API expiry returns 302, not 401.** `GranCRMSessionMiddleware` redirects
  invalid/expired `grancrm_session` cookies to login *before* the ninja API runs,
  so `/incitrack/api/*` returns `302`→login-HTML on expiry instead of `401`. The
  React `apiFetch` (§2.2.3) expects `401` to emit `grancrm:sessionExpired`. In
  Plan 003/004, exempt `/incitrack/api/` from the redirecting middleware (or make
  it return `401` for API paths) so the SPA handles session expiry correctly.
  The API itself already returns `401` when no cookie is present.
  **RESOLVED in 004-prep** (`exec/incitrack-004`): `GranCRMSessionMiddleware` now
  returns `401 JSON` for `/incitrack/api/` paths on invalid/expired session and
  keeps the `302` redirect for non-API paths (test-covered).

- **Minor parity gap (003):** the React user-edit form does not port the Django
  `UsuarioEditarForm.supervisores_asignados` *jefe* workflow (assign supervisors
  under a jefe-de-cuenta, which adds them to the jefe's cuentas' `supervisores`).
  Not a security issue (no mis-wiring); the same `cuenta.supervisores` M2M is fully
  manageable via the Cuenta form's supervisores list and the per-supervisor's
  "cuentas asignadas" — both of which ARE ported. Add the jefe convenience form
  only if strict parity is wanted.

## Cross-cutting requirements (the user emphasized these)

- **Nativo**: real React components + JSON API. No iframe, no `?embed=1`, no
  server-rendered HTML for the app pages.
- **Responsivo**: the UI must work on narrow viewports. Use the shell's Bootstrap
  grid / flex utilities (already loaded by the shell theme) and avoid fixed pixel
  layouts. Verify at ≤768px.
- **Portable**: all paths **relative** (no `http://IP:port` anywhere). API calls
  derive from the `apiBase` prop (guide §2.2.3 / MEDIO-1), routes use the
  `basename` the shell injects. The bundle must not hardcode the host.
